# WBS / Gantt 입력 자동화 기획안

## 1. 문제 정의

현재 `DK Flow`의 WBS와 Gantt는 직접 입력 중심이다.

- 작업을 한 줄씩 추가해야 한다.
- 상위/하위 구조를 사람이 직접 잡아야 한다.
- 일정도 작업별로 시작일/종료일을 직접 넣어야 한다.
- 반복 프로젝트여도 이전 구조를 재사용하기 어렵다.

이 방식은 기능적으로는 충분하지만, 실제 사용자는 `작성`보다 `수정`에 집중하고 싶어한다.
따라서 목표는 입력 방식을 `수기 등록`에서 `초안 생성 후 검토`로 바꾸는 것이다.

## 2. 목표

### 사용자 목표

- 프로젝트 초안을 1분 안에 만들 수 있어야 한다.
- 작업 구조를 0에서 작성하지 않고 추천받아야 한다.
- 날짜를 하나씩 넣지 않고 규칙으로 계산해야 한다.
- 이전 프로젝트나 엑셀 데이터로 빠르게 시작할 수 있어야 한다.

### 제품 목표

- 기존 WBS 편집기와 Gantt를 유지한 채 입력 효율만 올린다.
- AI 의존 없이도 동작하는 규칙 기반 자동화를 우선 제공한다.
- 이후 자연어 생성이나 외부 LLM 연동으로 확장 가능한 구조를 만든다.

## 3. 제안 기능 구조

자동화 기능은 아래 4개로 나눈다.

### A. 템플릿 기반 WBS 생성

프로젝트 유형을 고르면 기본 WBS를 자동 생성한다.

예시 유형:

- 웹사이트 구축
- 모바일 앱 개발
- 운영/유지보수
- 마케팅 캠페인
- 사내 시스템 구축

생성 항목:

- Phase / Activity / Task 기본 계층
- 기본 산출물명
- 기본 duration
- 기본 선후행 규칙

### B. 빠른 입력 생성기

사용자가 짧은 설명만 입력하면 작업 초안을 생성한다.

예:

`쇼핑몰 리뉴얼 / 6월 오픈 / 디자인, 프론트, 백엔드, QA`

생성 결과:

- 상위 작업 구조
- 하위 작업 후보
- 역할별 담당 추천
- 초기 일정 계산값

MVP에서는 LLM 없이 키워드 매핑 + 템플릿 조합으로 구현 가능하다.

### C. 일정 자동 계산

사용자가 날짜를 전부 넣지 않게 하고, 아래 입력만 받는다.

- 프로젝트 시작일
- 작업 duration
- 선행 작업

그러면 자동으로:

- planStart
- planEnd
- 상위 작업 기간 roll-up

을 계산한다.

### D. 가져오기 / 재사용

기존 데이터에서 초안을 만든다.

- 이전 프로젝트 복제
- Excel 업로드 후 자동 매핑
- 추후 Notion / Jira 연동 가능

현재 코드베이스에 `src/lib/excel.ts`가 있으므로, 가져오기는 내보내기 구조를 재활용하는 방식이 적합하다.

## 4. MVP 범위

가장 먼저 넣을 범위는 아래 3개다.

### MVP-1. 템플릿 생성

- 프로젝트 생성 또는 WBS 화면에서 `템플릿으로 시작` 버튼 제공
- 템플릿 선택 후 기본 작업 구조 생성
- 생성 후 기존 WBS 표에서 바로 수정 가능

### MVP-2. 빠른 입력 생성

- WBS 상단에 `빠른 초안 생성` 입력창 추가
- 한 줄 설명 입력
- 키워드 분석 후 템플릿 + 추가 작업 추천
- 결과를 미리보기 모달에서 확인 후 반영

### MVP-3. 일정 자동 계산

- Task에 `durationDays`, `predecessorIds` 개념 추가
- `자동 일정 계산` 버튼 제공
- planStart/planEnd 자동 입력
- 상위 작업 기간 자동 roll-up

이 세 가지면 사용자는 거의 모든 신규 프로젝트를 `생성 후 수정` 방식으로 다룰 수 있다.

## 5. 화면 흐름 제안

### 5.1 프로젝트 생성 시점

`프로젝트 생성 > 시작 방식 선택`

- 빈 프로젝트
- 템플릿으로 시작
- 이전 프로젝트 복제
- 엑셀 가져오기

### 5.2 WBS 화면

현재 `src/pages/WBS.tsx` 상단 액션 영역에 아래 액션을 추가한다.

- `빠른 초안 생성`
- `템플릿 적용`
- `자동 일정 계산`

권장 흐름:

1. 템플릿 또는 빠른 입력으로 초안 생성
2. 미리보기에서 작업 추가/삭제
3. WBS 반영
4. 자동 일정 계산
5. 사용자가 세부 조정

### 5.3 Gantt 화면

현재 `src/pages/Gantt.tsx`는 편집과 조회가 강하다.
여기에는 생성보다 결과 검토 기능을 붙이는 편이 맞다.

- 자동 계산 결과 강조
- 날짜 충돌 / 지연 위험 표시
- 선행 작업 위반 표시

## 6. 데이터 모델 변경안

현재 `Task` 타입은 직접 입력형 구조다. 자동화를 위해 아래 필드가 필요하다.

추가 후보:

- `templateId?: string`
- `durationDays?: number | null`
- `predecessorIds?: string[]`
- `autoScheduled?: boolean`
- `taskSource?: 'manual' | 'template' | 'quick_draft' | 'imported'`

적용 위치:

- 타입: `src/types/index.ts`
- 저장소: `src/store/taskStore.ts`
- 동기화/정규화: `src/lib/projectTaskSync.ts`
- 저장 계층: `src/lib/dataRepository.ts`

참고:
현재 DB 스키마에는 `tasks` 테이블에 duration/predecessor 개념이 없다. 로컬 저장만 먼저 지원하고, 이후 Supabase migration으로 확장하는 순서가 안전하다.

## 7. 모듈 설계 제안

새 모듈은 아래처럼 나누는 것이 맞다.

### `src/lib/taskTemplates.ts`

역할:

- 프로젝트 유형별 템플릿 정의
- 템플릿 목록 조회
- 템플릿 기반 Task 배열 생성

### `src/lib/taskDraft.ts`

역할:

- 빠른 입력 문장 파싱
- 키워드 기반 템플릿 매칭
- 추가 작업 후보 생성

### `src/lib/taskScheduler.ts`

역할:

- 선행 작업 기준 일정 계산
- duration 기반 시작/종료일 산출
- 상위 작업 기간 roll-up

### `src/components/wbs/TaskGenerationModal.tsx`

역할:

- 템플릿 선택
- 빠른 입력
- 생성 결과 미리보기
- 반영 여부 확인

## 8. 생성 로직 제안

### 8.1 템플릿 생성

입력:

- projectType
- projectStartDate
- optional team size

출력:

- `Task[]`

규칙:

- 루트는 Phase
- 하위는 Activity / Task / Function
- sibling order 자동 부여
- 기본 duration 삽입

### 8.2 빠른 입력 생성

1. 문장에서 키워드 추출
2. 프로젝트 유형 점수화
3. 가장 가까운 템플릿 선택
4. 누락된 키워드 기반 작업 추가
5. 초안 반환

예:

- `QA`, `테스트` 포함 -> 테스트 Activity 추가
- `디자인` 포함 -> 디자인 하위 작업 추가
- `오픈`, `배포` 포함 -> 배포/안정화 단계 추가

### 8.3 일정 계산

규칙:

- predecessor가 없으면 프로젝트 시작일 기준
- predecessor가 있으면 가장 늦은 종료일 다음날 시작
- durationDays 기준으로 종료일 계산
- 부모는 자식 min(start), max(end)로 계산

MVP에서는 휴일/주말 제외 계산 없이 달력 기준 일수로 시작하고, 이후 옵션화한다.

## 9. 우선순위

### Phase 1

- 템플릿 생성
- 빠른 입력 생성
- 자동 일정 계산

### Phase 2

- 이전 프로젝트 복제 강화
- Excel 가져오기 매핑 UI
- 생성 결과 diff 미리보기

### Phase 3

- LLM 기반 자연어 생성
- Notion / Jira 연동
- 조직별 커스텀 템플릿 관리

## 10. 구현 순서 제안

### Step 1. 템플릿 엔진

- `taskTemplates.ts` 추가
- 2~3개 대표 템플릿 정의
- WBS에서 템플릿 적용 버튼 연결

### Step 2. 일정 계산기

- `taskScheduler.ts` 추가
- duration/predecessor 필드 반영
- WBS/Gantt에 계산 결과 표시

### Step 3. 빠른 입력 생성기

- `taskDraft.ts` 추가
- 키워드 파서 작성
- 생성 미리보기 모달 추가

### Step 4. 저장 구조 확장

- localStorage 저장 포맷 확장
- 이후 Supabase migration 추가

## 11. 성공 지표

- 신규 프로젝트 생성 후 첫 WBS 완성까지 걸리는 시간
- 사용자가 직접 추가한 작업 수 대비 자동 생성 작업 수
- 템플릿 사용률
- 빠른 입력 생성 후 수정률
- 자동 일정 계산 사용률

핵심 지표는 `직접 입력 행 수 감소`다.

## 12. 권장 결론

이 프로젝트에서는 처음부터 AI 챗 기반 자동 생성을 크게 붙이는 것보다, 아래 순서가 맞다.

1. 템플릿 기반 생성
2. 규칙 기반 빠른 입력 생성
3. 선행 작업 기반 자동 일정 계산
4. 이후 LLM 연동

이유는 현재 코드베이스가 이미 `WBS 편집기`, `Gantt`, `chatbot`, `excel`, `zustand store`를 갖추고 있어서, 입력 자동화만 얹으면 바로 체감 가치가 나오기 때문이다.

즉, `새 시스템 개발`이 아니라 `기존 편집 흐름의 초안 생성기 추가`로 접근하는 것이 가장 효율적이다.
